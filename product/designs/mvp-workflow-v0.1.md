# MVP Workflow Design v0.1

> **Scope**: 定义 MVP 端到端工作流——从用户输入到接近可投版简历输出的完整链路。  
> **Status**: Draft.  
> **Target Output**: Level 2（接近可投版），需用户确认后才成为 Level 3（确认投递版）。

---

## 0. 工作流总览

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  阶段 0     │───→│  阶段 1     │───→│  阶段 2     │───→│  阶段 3     │
│ 用户输入    │    │ JD 解析     │    │ 素材结构化  │    │ 证据映射    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                                                              │
                                                              ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  阶段 7     │←───│  阶段 6     │←───│  阶段 5     │←───│  阶段 4     │
│ 最终输出    │    │ 用户确认    │    │ 真实性校验  │    │ 受约束生成  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**工作流核心原则**：

1. **不可逆修改必须留痕**：任何对原文的改写、升级、降级都必须记录理由
2. **无证据不生成**：每条简历 bullet 必须能追溯到至少一个素材事实
3. **弱证据弱表达**：证据强度决定表达强度，不能为了"好看"而升级
4. **高风险自动降级**：当 JD 与素材差距大时，系统进入保守模式
5. **用户是最终守门人**：系统输出的是"接近可投版"，用户确认后才算完成

## 1. 阶段 0：用户输入采集

### 1.1 用户输入什么

MVP 阶段用户需要提供三类信息：

#### A. 基础身份信息

| 字段 | 必填 | 说明 |
|---|---|---|
| 姓名 | 是 | |
| 邮箱 | 是 | |
| 电话 | 否 | |
| GitHub/博客 | 否 | 如有则填入 |
| 所在城市 | 否 | 影响到岗时间判断 |

#### B. 目标岗位

| 字段 | 必填 | 说明 |
|---|---|---|
| 公司名 | 是 | |
| 岗位名称 | 是 | |
| JD 原文 | 是 | **完整粘贴**，不要摘要 |
| 招聘页面 URL | 否 | 用于溯源和自动补充 |

> **为什么要求完整 JD？**  
> Baseline 测试发现，摘要 JD 会丢失大量语义信息（如"对 AI 产品的交互设计、能力边界等问题有自己独到的思考"这类抽象要求）。完整 JD 是后续解析的基础。

#### C. 原始素材包

用户以**文本块**形式提供，每块一个经历/素材。

**素材类型**：

| 类型 | 示例 | 重要性 |
|---|---|---|
| 教育背景 | 学校、专业、毕业时间 | 必须提供 |
| 项目经历 | 课程项目、个人项目、科研项目 | 核心素材 |
| 竞赛经历 | 黑客松、算法竞赛、产品大赛 | 高价值素材 |
| 校园经历 | 社团、学生会、志愿者 | 可补充素材 |
| 技能自评 | 会什么工具/语言/框架 | 用于硬条件匹配 |
| 其他 | 任何觉得相关的经历 | 系统会判断是否有用 |

**素材格式要求（对用户）**：

- 不需要结构化，口语化描述即可
- 建议包含：做了什么、用了什么、结果如何
- 时间范围如果有就写，没有也行
- **诚实比完整更重要**：没做过的事不要写

### 1.2 输入校验

系统在接收输入时做**轻量校验**：

| 校验项 | 不通过时的处理 |
|---|---|
| JD 是否过短（<50 字） | 提示用户"JD 可能不完整，建议从招聘页面完整复制" |
| 素材是否为空 | 拒绝生成，提示"请至少提供教育背景和一段项目经历" |
| 素材是否全是技能列表 | 提示"建议补充具体项目或经历描述，仅有技能列表难以生成高质量简历" |
| 基础信息是否完整 | 标记缺失项，但不阻断流程 |

### 1.3 输出物

- `UserInput` 对象（见 `data-structures-v0.1.md` §1）

## 2. 阶段 1：JD 解析

### 2.1 目标

将非结构化的 JD 原文，解析为结构化的岗位要求图谱。

### 2.2 JD 如何解析

解析过程分为三层：

#### 层 1：文本分块与分类

将 JD 原文按语义切分为段落，并分类：

| 段落类型 | 示例 | 处理策略 |
|---|---|---|
| 岗位描述 | "参与实时模型的数据梳理和整理..." | 提取动作和能力暗示 |
| 硬性要求 | "本科及以上相关学历在读" | 归入 HardRequirement |
| 核心能力 | "具备市场敏锐度和信息收集能力" | 归入 CapabilityRequirement（critical） |
| 加分项 | "对 AI 工具化产品和大模型迭代有相关经验者优先" | 归入 CapabilityRequirement（nice_to_have） |
| 其他信息 | "我们提供：与行业领先的 AI 团队一起工作..." | 提取文化信号，不入要求列表 |

#### 层 2：要求抽取与结构化

对每类段落进行细粒度抽取：

**硬门槛抽取**：
- 识别学历、专业、年级、到岗时间、实习时长、语言要求
- 识别特定工具/软件要求（Python、SQL、Figma 等）
- 输出为 `HardRequirement` 列表

**能力要求抽取**：
- 将抽象描述转化为结构化能力项
- 例如："对海内外 AI 模型及相关产品有基础认知，使用过至少两款 AIGC 产品"
  - 能力：AI 产品认知
  - 证据类型：product_judgment
  - 关键词：["AIGC 产品", "AI 模型", "产品认知"]
- 输出为 `CapabilityRequirement` 列表，标注优先级

#### 层 3：岗位上下文推断

基于 JD 整体内容推断：

| 推断项 | 用途 |
|---|---|
| 职级（实习/初级/中级） | 决定简历的叙事策略（实习生应强调学习能力和潜力） |
| 团队方向 | 决定哪些技能标签更重要 |
| 产品阶段 | 早期团队更看重探索能力，成熟团队更看重执行力 |
| 文化信号 | 如"快节奏"→强调快速学习能力；"研究驱动"→强调分析深度 |

### 2.3 解析质量控制

| 检查项 | 标准 | 未达标处理 |
|---|---|---|
| 硬门槛覆盖率 | ≥90% 的硬门槛被识别 | 低置信度时标记"建议人工复核" |
| 能力项数量 | 3-10 个核心能力项 | 过少时检查是否漏抽；过多时合并相似项 |
| 原文溯源 | 每个抽取项必须关联原文片段 | 无法溯源的项标记为"推断" |

### 2.4 输出物

- `JDParsedResult` 对象（见 `data-structures-v0.1.md` §2）
- 附：解析置信度评分，低于阈值时提示用户"JD 解析可能不完整，建议检查"

## 3. 阶段 2：素材结构化

### 3.1 目标

将用户提供的口语化、非结构化原始素材，转化为可计算、可映射的结构化事实。

### 3.2 原始素材如何结构化

#### 步骤 1：素材分类与初步清洗

系统根据素材标题和内容自动分类：

| 分类依据 | 处理方式 |
|---|---|
| 包含"大学""学院""专业""GPA"等词 | 标记为 education |
| 包含"项目""开发""设计""实现"等词 + 技术栈 | 标记为 project |
| 包含"比赛""竞赛""黑客松""获奖" | 标记为 competition |
| 包含"社团""学生会""志愿者""组织" | 标记为 campus |
| 仅包含技能列表（Python、SQL 等） | 标记为 skill |

#### 步骤 2：事实抽取

从每段素材中抽取最小事实单元 `MaterialFact`：

**抽取维度**：

| 维度 | 示例 | 说明 |
|---|---|---|
| 动作事实 | "整理过知识库内容" | 用户做过什么 |
| 成果事实 | "对比过不同 prompt 的回答差异" | 产生了什么结果 |
| 技能事实 | "会 Python" | 掌握/使用过什么 |
| 认知事实 | "对 AI 产品市场趋势有自己的简单判断" | 具备什么认知 |
| 特质事实 | "对市场变化敏感" | 具备什么特质 |

**置信度标注**：

| 置信度 | 判定标准 | 示例 |
|---|---|---|
| explicit | 用户明确陈述 | "我负责整理知识库" |
| inferred_strong | 强推断，素材明确支持 | "独立开发了一个问答助手"→solo |
| inferred_weak | 弱推断，需谨慎使用 | "参加过这个项目"→可能参与过协作 |

#### 步骤 3：标签生成

为每个事实自动生成标签，用于后续映射：

- **skillTags**: Python、Figma、Prompt Engineering、数据分析...
- **topicTags**: Agent、RAG、AIGC、产品评测、知识库...
- **outcomeTags**: 原型、报告、对比分析、Bad Case 整理...

#### 步骤 4：解析注释

系统生成 `ParserNote`，标记需要用户注意的问题：

| 注释级别 | 示例 | 处理 |
|---|---|---|
| info | "素材中未提及具体时间" | 记录，不阻断 |
| warning | "'做过一个问答助手'描述较模糊，建议补充具体功能和技术栈" | 在确认阶段展示 |
| critical | "素材中出现明显矛盾（前说'独立完成'，后说'团队项目'）" | 在确认阶段高亮 |

### 3.3 输出物

- `MaterialParseResult` 对象（见 `data-structures-v0.1.md` §3）
- 附：解析注释列表，供后续用户确认时参考

## 4. 阶段 3：证据映射

### 4.1 目标

建立 JD 要求 ↔ 素材事实 之间的映射关系，回答三个问题：

1. 哪些 JD 要求**有证据支撑**？
2. 哪些 JD 要求**证据不足**？
3. 哪些素材事实**未被 JD 利用**？

### 4.2 JD 要求如何映射到素材证据

#### 映射策略

系统采用三级映射策略，按优先级尝试：

| 映射级别 | 名称 | 逻辑 | 示例 |
|---|---|---|---|
| L1 | 直接映射 | 关键词/技能直接匹配 | JD:"会 Python" → 素材:"会 Python" |
| L2 | 语义映射 | 不同表述但语义相同 | JD:"数据品味" → 素材:"整理 bad case 并分类归因" |
| L3 | 推断映射 | 需要推理链条支撑 | JD:"产品判断能力" → 素材:"做过竞品功能对比并给出评价" |

#### 映射过程

对每个 `CapabilityRequirement`，系统执行：

1. **候选事实检索**：基于 skillTags / topicTags / outcomeTags 召回相关 `MaterialFact`
2. **相关性评分**：计算 JD 要求与候选事实的语义相似度
3. **证据强度判定**：
   - 强证据：直接匹配 + explicit 置信度
   - 中等证据：语义匹配 或 强推断
   - 弱证据：推断匹配 或 弱推断
   - 证据不足：无相关事实
4. **映射理由生成**：为每个映射生成自然语言解释（`reasoning`）

#### 组合映射

当多个弱事实共同支撑一个要求时，系统可创建 `composite` 映射：

> **示例**：
> - JD 要求："对创新产品有独特见解"
> - 素材事实 1："对每款 AIGC 产品做过简单功能对比"
> - 素材事实 2："对 GLM 系列模型有持续关注"
> - 素材事实 3："对 AI 产品市场趋势有自己的简单判断"
> - 映射类型：composite
> - 证据强度：moderate（多个弱证据组合）
> - 理由："用户虽无正式产品分析经验，但具备持续关注、主动对比、独立判断的习惯，可支撑'对创新产品有独特见解'的潜力描述"

### 4.3 Gap 识别

对无证据或证据不足的 JD 要求，生成 `GapItem`：

| Gap 类型 | 处理策略 |
|---|---|
| missing_evidence | 标记为"未满足"，建议用户补充或在简历中回避 |
| insufficient_depth | 使用保守表达，如"接触过..."而非"精通..." |
| unclear_scope | 在确认阶段询问用户澄清 |
| temporal_mismatch | 如实标注时间，不虚构延长 |

### 4.4 输出物

- `EvidenceMappingResult` 对象（见 `data-structures-v0.1.md` §4）
- 附：Gap 列表、Overclaim 列表

## 5. 阶段 4：受约束生成

### 5.1 目标

在真实性约束下生成简历草稿，核心约束：**禁止无证据植入、禁止夸大、禁止新增事实**。

### 5.2 哪些内容可以写进简历

#### 准入规则

内容进入简历必须满足以下条件之一：

| 条件 | 说明 |
|---|---|
| 有强证据支撑 | 至少一个 `EvidenceMapping` 的 strength ≥ moderate |
| 属于基础信息 | 姓名、学校、专业等客观事实 |
| 属于硬条件自证 | 用户明确声明的学历、技能、语言能力 |
| 用户明确要求保留 | 用户标记"这段一定要写"的素材 |

#### 优先级排序

当素材较多时，按以下优先级决定写入顺序和详略：

1. **高匹配 + 强证据**：与 critical JD 要求直接对应，且有强证据
2. **高匹配 + 中等证据**：与 critical/important JD 要求对应，证据中等
3. **硬条件相关**：学历、技能等硬门槛证明
4. **中等匹配 + 强证据**：与 nice_to_have 对应，但证据强
5. **展示成长潜力**：无直接 JD 对应，但能展示学习能力的经历
6. **其他素材**：剩余内容酌情放入"补充信息"

### 5.3 哪些内容必须降级

#### 降级触发条件

| 触发条件 | 降级策略 | 示例 |
|---|---|---|
| 弱证据（weak） | 使用保守动词和程度词 | "主导"→"参与"；"精通"→"了解" |
| 推断事实（inferred_weak） | 降级为 literal 表达或删除 | "可能参与过协作"→不写 |
| 时间模糊 | 不标注具体时间，或标注"在校期间" | |
| 范围不明确 | 使用模糊量词或不写数量 | "做过一些测试"→保留原样，不升级为"完成 50+ 测试用例" |
| 课程项目 | 明确标注"课程项目"，不包装成实习 | |
| 无直接 JD 对应 | 简短提及或放入补充信息 | |

#### 表达强度与证据强度的对应关系

```
Evidence Strength      Expression Level
─────────────────────────────────────────
strong        →       standard / emphasized（可选）
moderate      →       standard
weak          →       conservative
insufficient  →       不写入 / literal（仅陈述存在）
```

### 5.4 生成流程

#### 步骤 1：章节规划

根据岗位类型和素材分布，决定简历章节结构：

**AI 产品岗标准结构**：
1. 基本信息
2. 教育背景
3. 项目经历（核心，2-3 个）
4. 技能/工具
5. 补充信息（竞赛、校园经历等）

**AI 应用开发岗标准结构**：
1. 基本信息
2. 教育背景
3. 项目经历（技术项目优先，2-3 个）
4. 技能/技术栈
5. 补充信息

#### 步骤 2：Bullet 生成

每个 bullet 的生成遵循"证据 → 表达"链条：

1. 选取一个 `EvidenceMapping`
2. 根据 evidence strength 确定 expression level
3. 从 `MaterialFact.statement` 出发，改写为简历表达
4. 记录 `RewriteStep`（原话 → 改写后 → 理由）
5. 附加 `EvidenceRef`（溯源信息）

#### 步骤 3：风险自检

生成后立即执行轻量风险检查：

| 检查项 | 发现后的处理 |
|---|---|
| 动词是否被升级 | 自动降级（"负责"→"参与"） |
| 是否出现具体数字 | 检查数字是否有素材来源，无来源则删除或模糊化 |
| 是否新增技能 | 检查技能是否在素材 skillTags 中，不在则删除 |
| 是否暗示实习经历 | 检查是否有 work 类型素材支撑，无则标注"课程项目/个人项目" |

### 5.5 输出物

- `ResumeDraft` 对象（见 `data-structures-v0.1.md` §5）
- 附：生成日志、风险标记

## 6. 阶段 5：真实性校验

### 6.1 目标

对生成的简历草稿进行多维度校验，拦截虚构、夸大、误导性包装。

### 6.2 如何避免虚构和过度包装

#### 校验维度

**维度 1：真实性检查（Authenticity Check）**

| 检查项 | 方法 | 阈值 |
|---|---|---|
| 事实一致性 | 每条 bullet 与关联的 MaterialFact 对比 | 不得出现素材中无的事实 |
| 数字溯源 | 提取 bullet 中的数字，检查是否在素材中 | 无来源的数字标记为风险 |
| 技能一致性 | bullet 中的技能词 vs 素材 skillTags | 不得出现素材未提及的技能 |
| 角色一致性 | bullet 中的角色描述 vs 素材 roleIndicator | 不得升级角色（participant→lead） |

**维度 2：岗位匹配度检查（JD Alignment Check）**

| 检查项 | 方法 | 阈值 |
|---|---|---|
| Critical 要求覆盖率 | 统计 critical JD 要求中有 mapping 的比例 | ≥60% 为通过 |
| 关键词自然度 | 检查 JD 关键词是否以生硬方式植入 | 避免"为了匹配而匹配"的表达 |
| 定制深度 | 检查内容是否围绕该 JD 特有要求组织 | 排除通用模板感 |

**维度 3：表达质量检查（Expression Quality Check）**

| 检查项 | 方法 | 阈值 |
|---|---|---|
| 简历口语 | 检测非正式表达 | 不得出现口语化用词 |
| bullet 结构 | 检查是否遵循"动作 + 方法 + 结果"结构 | 建议遵循，非强制 |
| 重复度 | 检查不同 bullet 是否重复描述同一事实 | 重复度 <30% |

**维度 4：结构完整度检查（Structural Completeness）**

| 检查项 | 方法 | 阈值 |
|---|---|---|
| 必含章节 | 检查基本信息、教育背景、项目经历是否存在 | 必须存在 |
| bullet 数量 | 统计总 bullet 数 | 6-12 条为合适范围 |
| 信息密度 | 检查是否有过长/过短的 bullet | 建议 1-2 行/条 |

**维度 5：修改成本估算（Modification Cost Estimate）**

| 检查项 | 方法 | 阈值 |
|---|---|---|
| 明显错误数 | 统计需用户修改的明显问题 | ≤3 处为低修改成本 |
| 模糊表达数 | 统计需用户澄清的模糊内容 | ≤5 处为可接受 |
| 压缩需求 | 检查是否过长需用户手动压缩 | 过长时系统主动建议删减 |

#### 红线拦截机制

以下情况触发**强制拦截**，必须修正后才能进入下一阶段：

| 红线类型 | 判定标准 | 系统处理 |
|---|---|---|
| 严重虚构 | bullet 中出现素材完全不存在的经历/成果/技能 | 删除该 bullet，标记为"无证据" |
| 角色升级 | 将"参与"写成"主导""负责" | 降级动词 |
| 时间虚构 | 捏造实习时长、项目周期 | 删除时间或标注"未明确" |
| 指标推断 | 无来源地添加具体数字（"提升 30%"） | 删除数字或改为模糊表达 |
| 课程包装 | 将课程作业描述为正式实习/工作 | 明确标注"课程项目" |
| 关键词植入 | 为匹配 JD 而无证据地添加技能关键词 | 删除无证据关键词 |

### 6.3 校验结果处理

| 结果 | 处理 |
|---|---|
| 全部通过 | 进入阶段 6（用户确认） |
| 有建议修改项 | 进入阶段 6，同时在确认界面展示建议 |
| 有必须修改项 | 自动修订后重新校验，仍不通过则标记为"高风险草稿" |
| 多个必须修改项 | 触发保守模式：系统生成更保守的版本 |

### 6.4 输出物

- `ValidationResult` 对象（见 `data-structures-v0.1.md` §6）
- 附：必须修改项列表、建议修改项列表

---

## 7. 阶段 6：用户确认

### 7.1 目标

让用户成为最终守门人。系统展示生成逻辑和证据来源，用户决定接受、修改或拒绝每条内容。

### 7.2 哪些内容需要用户确认

#### 必须确认的内容（高优先级）

| 内容类型 | 原因 |
|---|---|
| 所有 `RiskLevel = warning` 的 bullet | 存在较大推断或模糊性 |
| 所有 `ExpressionLevel = emphasized` 的 bullet | 系统做了升级表达 |
| 所有涉及角色描述的 bullet | 容易误升级（"参与"→"负责"） |
| 所有包含数字的 bullet | 数字最容易被虚构或推断 |
| 所有 gap 项 | 用户需决定是否接受"能力缺口"的披露 |

#### 建议确认的内容（中优先级）

| 内容类型 | 原因 |
|---|---|
| `ExpressionLevel = standard` 的 bullet | 标准表达，但建议用户核对 |
| 组合映射（composite）的 bullet | 多个弱证据组合，需用户判断是否成立 |
| 放入"补充信息"的素材 | 确认是否值得保留 |

#### 自动通过的内容（低优先级）

| 内容类型 | 原因 |
|---|---|
| 基础信息（姓名、学校、专业） | 客观事实，用户已提供 |
| `RiskLevel = safe` + `ExpressionLevel = literal` 的 bullet | 几乎原文转述，风险极低 |

### 7.3 确认界面设计要点

#### 展示内容

对每个需确认的 bullet，界面展示：

1. **当前文本**：系统生成的简历表达
2. **证据来源**：用户原始素材中的相关片段（高亮显示）
3. **映射理由**：系统为什么认为这个素材能支撑这个 JD 要求
4. **风险提示**：如果有风险，展示具体风险类型和系统建议
5. **系统建议**："建议保留"/"建议修改"/"建议删除"

#### 用户操作

| 操作 | 结果 |
|---|---|
| ✅ 认可 | bullet 进入最终简历，标记为 user approved |
| ✏️ 修改 | 用户编辑文本，系统记录修改前后差异 |
| ❌ 拒绝 | bullet 被删除，系统询问是否用其他素材替代 |
| 💬 备注 | 用户添加备注，不修改文本 |

#### Gap 确认

对 `GapItem`，界面展示：

1. **未满足的 JD 要求**："岗位要求：具备数据分析能力"
2. **当前状态**："素材中未找到相关证据"
3. **用户选项**：
   - "接受缺口"：在简历中不提及该能力
   - "后续补充"：用户承诺补充素材后重新生成
   - "我有但忘写了"：用户补充素材，系统重新映射

### 7.4 确认会话结束条件

满足以下任一条件即可结束确认：

1. 所有高优先级项已处理（认可/修改/拒绝）
2. 用户主动点击"完成确认"
3. 用户拒绝的 bullet 数量 > 总 bullet 数量的 50%（触发重新生成建议）

### 7.5 输出物

- `ConfirmationSession` 对象（见 `data-structures-v0.1.md` §7）
- 附：用户决策日志、gap 确认记录

---

## 8. 阶段 7：最终输出

### 8.1 目标

整合用户确认后的结果，输出"接近可投版"简历及辅助文档。

### 8.2 最终输出物是什么

#### 主输出：接近可投版简历

格式：Markdown（后续可扩展为 PDF/DOCX）

内容：
- 用户确认后的 `ResumeDraft`
- 结构：基本信息 → 教育背景 → 项目/经历 → 技能 → 补充信息
- 风格：标准中文实习简历格式

#### 附件 1：证据映射表

用途：让用户理解"为什么简历这样写"

内容：
| JD 要求 | 简历表达 | 素材来源 | 证据强度 |
|---|---|---|---|
| 具备市场敏锐度 | 持续关注近半年 AI 产品发布动态，整理 5-6 款主流产品功能更新 | 素材 3：市场信息整理 | moderate |

#### 附件 2：Gap 报告

用途：让用户知道"我还有哪些没满足"

内容：
- 未满足的 JD 要求列表
- 严重程度标注
- 用户选择的处理方式（接受/后续补充）

#### 附件 3：风险提示摘要

用途：投递前最后检查

内容：
- 简历中仍存在的 caution/warning 级别风险
- 系统建议的修改方向
- 用户已覆盖的 redline 项（如有）

#### 附件 4：修改建议指南

用途：帮助用户从"接近可投版"进化为"确认投递版"

内容：
- 如何进一步压缩长度
- 如何根据具体面试官调整侧重点
- 哪些 gap 可以在面试中主动解释

### 8.3 输出层级声明

系统在输出顶部明确声明：

> **本简历为"接近可投版"（Level 2），已由系统生成并经过初步校验，但最终投递前请您：**
> 1. 核对所有事实是否与您的真实经历一致
> 2. 确认所有数字、时间、技能描述准确无误
> 3. 根据目标岗位微调侧重点
> 4. **最终投递版（Level 3）的责任由您承担**

### 8.4 输出物

- `ResumeOutput` 对象（见 `data-structures-v0.1.md` §8）

## 9. 异常处理与分支流程

### 9.1 素材严重不足

**判定**：可用 MaterialFact < 3 个，或无任何 project 类型素材

**处理**：
1. 系统进入"保守模式 + Gap 披露"
2. 生成极简简历（仅基础信息 + 教育背景 + 技能列表）
3. 向用户说明："当前素材不足以生成完整项目经历，建议补充以下素材..."
4. 提供素材补充建议清单

### 9.2 JD 与素材严重不匹配

**判定**：critical JD 要求的 evidence coverage < 30%

**处理**：
1. 系统生成简历，但大幅降级表达
2. 生成详细 Gap 报告
3. 向用户说明："目标岗位与您的素材存在一定差距，已采用保守表达策略"
4. 建议用户：调整目标岗位 或 补充匹配素材

### 9.3 用户大量拒绝

**判定**：用户拒绝的 bullet 数量 > 50%

**处理**：
1. 询问用户拒绝原因（"表达不准确"/"素材理解有误"/"我不想要这条"）
2. 如果是"素材理解有误"：返回阶段 3 重新映射
3. 如果是"表达不准确"：返回阶段 4 重新生成（带用户反馈）
4. 如果是"我不想要这条"：删除并尝试用其他素材替代

### 9.4 校验不通过且无法自动修复

**判定**：存在 redline 项，系统自动降级后仍不通过

**处理**：
1. 不输出简历，向用户展示问题
2. 说明："系统在以下 bullet 中检测到高风险内容，无法自动生成安全版本"
3. 提供两个选项：
   - "查看详细问题并手动修正"
   - "删除问题 bullet 后生成精简版"

## 10. 工作流与 Baseline 结论的对应

| Baseline 发现的问题 | 本工作流的应对 |
|---|---|
| 通用 LLM 容易新增事实 | 阶段 5 真实性检查 + 阶段 6 用户确认 |
| 容易把弱素材写成强经历 | 阶段 4 表达强度与证据强度绑定 + 阶段 5 角色一致性检查 |
| 为匹配 JD 植入无证据关键词 | 阶段 5 技能一致性检查 + 阶段 3 证据映射 |
| 高风险样本最能暴露越界 | 阶段 4 保守模式 + 阶段 5 红线拦截 |
| ChatGPT 不能逐条证据追溯 | 全程 EvidenceRef 绑定 + 阶段 6 证据来源展示 |
| 缺少 JD 与素材的闭环映射 | 阶段 3 证据映射 + 附件 1 证据映射表 |
| 现有工具不支持从零生成 | 阶段 0-2 专门设计素材采集与结构化流程 |
| 中文实习场景支持不足 | 阶段 4 中文实习简历结构 + 阶段 7 场景适配 |
