# Judge 3 Blind Evaluation

## Persona
Career coach for undergraduate internship applicants. Prioritizes modification cost, clarity, and whether claims are defensible in interview.

## Scores

### A2
| Version | Job customization /25 | Authenticity /30 | Structure /15 | Expression /15 | Modification cost /15 | Total /100 | Fabrication detected? | Direct submit or minor edits? |
|---|---:|---:|---:|---:|---:|---:|---|---|
| Version 1 | 20 | 25 | 12 | 12 | 11 | 80 | No | Yes |
| Version 2 | 21 | 29 | 13 | 13 | 12 | 88 | No | Yes |

Reasons:
- Version 1: 与字节 AI 产品、开发者服务、Agent 和 AI Coding 方向有明确贴合，能把问答助手、Prompt 对比、Bad Case、JD 分析迁移到岗位需求；但部分表述如“适合申请”“具备一定技术理解能力”偏自评化，证据边界不如项目 bullet 清晰。
- Version 1: 真实性整体较稳，没有编造指标或正式实习，但把岗位信息整理小工具迁移到产品需求调研场景时略显包装，需要面试中解释项目规模和个人产出。
- Version 2: 明确标注【素材支撑】【合理推断】，主动披露缺少 AI Coding 产品、Agent 原理、跨团队协作、PRD 等能力，真实性和面试可防守性最好。
- Version 2: 结构更像“第一版简历 + Gap Report”，对投递前补充项很清楚；缺点是标注语和 Gap Report 不能原样投递，需要删除标注并补时间信息。
- Preferred version: Version 2 — 更保守、证据边界更清楚，能显著降低本科/硕士实习申请中最危险的夸大风险。

Reasons:
- Version 1: 对小红书 AI 应用开发、Agent、上下文理解、LLM 工程化落地等关键词匹配较好，同时把个人 Demo 的边界写清楚，没有把练习项目包装成生产级系统。
- Version 1: 真实性很强，明确列出“建议补充”“Missing JD Capabilities”和高风险 claims avoided，避免了 Multi-Agent、部署、智能代码生成等素材不足的硬凑；但正文中的【建议补充】不适合直接出现在简历里。
- Version 2: 更像可投递简历，表达自然，能清楚展示 Python API 调用、向量检索、引用不准、任务分解和工具调用实验，岗位相关性略强。
- Version 2: 虽未明显造假，但“项目与目标岗位中的 LLM 工程化、需求理解、代码辅助与研发工具链方向相关”略有拉伸，候选人需要能解释为什么文档问答 Demo 可迁移到研发工具链。
- Preferred version: Version 1 — 虽需删除标注和补项目细节，但对经验边界的控制更好，面试防守成本更低。

Reasons:

- Version 1: 没有明显编造正式产品经历或量化成果，且主动承认没有 benchmark、完整评测和正式产品经历；不过正文较长且自我评价式语句较多，需要压缩成更有证据的 bullet。
- Version 2: 真实性最强，几乎逐条承认材料缺口，避免主导、从 0 到 1、评测方案、多模态、量化提升等高风险 claims；从面试防守角度非常安全。
- Version 2: 简历正文过于保守，项目经历只有很少 bullet，且【素材支撑】【建议补充】和 Gap Report 不能直接投递；若不补作品集或评测小项目，对美团 AI PM 岗位竞争力不足。
- Preferred version: Version 2 — 真实性满分且风险控制更好，但需要较多修改和补充材料后才适合正式投递。

## Overall Judgment
- Which version generally performs better across samples: Mixed
- Does any version show systematic fabrication risk: Neither
- Practical hiring-screening recommendation: 候选人应优先从更保守、带有证据边界和 Gap Report 的版本开始，因为它能减少不 defensible 的面试风险；正式投递前需要删除内部标注、补时间/链接/项目规模，并把 Gap Report 中最关键的 1-2 个缺口转化为真实作品或补充材料。
