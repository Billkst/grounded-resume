# Judge 1 Scores

## Persona
互联网 AI 产品招聘经理，重视岗位匹配、可投递性和真实可信。

## A2

| Version | Job customization /25 | Authenticity /30 | Structure /15 | Expression /15 | Modification cost /15 | Total /100 | Fabrication detected | Direct submit/minor edits |
|---|---:|---:|---:|---:|---:|---:|---|---|
| Version 1 | 21 | 29 | 14 | 12 | 12 | 88 | No | Yes |
| Version 2 | 23 | 29 | 14 | 14 | 14 | 94 | No | Yes |

Preferred version: Version 2

Reasons:
- Version 1: 岗位方向明确写到字节开发者服务、AI Coding、Agent、开发者工具，匹配度较好。
- Version 1: 真实性较强，明确承认暂无正式实习，项目描述集中在知识库整理、问题分类、Prompt 对比和 Bad Case 分析，没有明显夸大指标。
- Version 1: 表达略偏“说明文”和自我评价式，部分内容重复，直接投递前需要压缩成更标准的项目 bullet。
- Version 2: 对豆包编程、Aime、AI Agent 平台、开发者工具的定位更精准，项目经历也更自然地迁移到需求分析和体验评估场景。
- Version 2: bullet 更简洁专业，结构更像可投递简历；同时保留“课程项目、暂无正式实习”的边界，可信度较高。
- Version 2: 修改成本最低，主要只需补充姓名联系方式和可能的时间信息即可投递。

Preferred version: Version 1

Reasons:
- Version 1: 与 Product Engineer（AI 应用方向）的匹配较强，突出 LLM 应用、RAG 原型、Agent、多轮提示词、工具调用和评估迭代。
- Version 1: 对项目规模边界描述清楚，说明是个人 Demo 和验证型练习，避免把练习包装成上线项目，真实可信。
- Version 1: 简历结构完整，项目标题和 bullet 都比较招聘友好，候选人补充个人信息后基本可直接投递。
- Version 2: 目标岗位和技能覆盖完整，也明确承认暂无大规模部署经验，整体可信。
- Version 2: 但格式更接近纯文本，项目 bullet 不如 Version 1 凝练；“Multi-Agent 协作、AI 自主交付”等表达略贴 JD，若原始经历支撑不足会显得有些包装过度。
- Version 2: 直接投递仍可用，但需要调整排版、压缩重复表述并弱化少量过强岗位术语。

## A6

Preferred version: Version 2

Reasons:
- Version 1: 非常真实克制，明确说明没有正式产品实习、没有系统 benchmark 或完整评测项目，对高风险稀疏素材的保真度很好。
- Version 1: 但弱点暴露过多，且多处以“后续希望补足”“目前尚未形成”收尾，作为投递简历会削弱招聘经理信心，修改成本较高。
- Version 1: 岗位匹配覆盖 Agent、Prompt、效果评测、产品优化，但项目成果感和产品经理可迁移能力表达偏弱。
- Version 2: 在不明显造假的前提下，把课程问答项目拆成输入、模型响应、结果判断、体验反馈等产品环节，更贴近美团 AI Agent 产品实习的工作语言。
- Version 2: 表达更凝练，能体现 Prompt 调试、效果观察、Agent 产品理解和移动端体验问题意识，稀疏素材下可投递性更强。
- Version 2: 仍保持经历边界，没有声称正式上线、量化指标或完整产品化交付；仅需补充联系方式和个别细节即可投递。
