# B2 Candidate Rerank

## Candidate Set

### Candidate 1：AI app product-engineering hybrid

- Framing：把文档问答 Demo 和任务分解工具整合为“AI 应用原型 + 产品工程流程”经历。
- Strength：同时覆盖 AI 应用、需求拆解、原型验证、评估迭代与 JD 语言，且不需要新增成果数字。
- Risk：需要控制“产品工程”表达，避免把个人练习夸大为正式交付。
- Score：Authenticity 29.5 / Readiness 29.0 / JD Fit 27.5 / Overall 86.0

### Candidate 2：Prototype / workflow / implementation collaboration

- Framing：强调原型、流程和实现协作语言。
- Strength：投递感较好，能弱化素材不足。
- Risk：原始素材没有团队协作，若过多使用 collaboration 容易显得虚。
- Score：Authenticity 28.5 / Readiness 28.0 / JD Fit 26.5 / Overall 83.0

### Candidate 3：User scenario + product requirement analysis

- Framing：突出用户场景、需求分析和产品表达。
- Strength：能提升 Product Engineer 定制感。
- Risk：素材主要是技术 Demo，产品需求文档、用户调研、原型图均未提供，过度产品化会降低真实性。
- Score：Authenticity 28.0 / Readiness 28.5 / JD Fit 26.0 / Overall 82.5

### Candidate 4：Technical product communication + evaluation iteration

- Framing：强调技术沟通、评估、问题发现和迭代意识。
- Strength：引用不准、效果不稳定等素材可自然转化为评估意识。
- Risk：技术产品沟通缺少多人协作证据，单独成稿会显得经历偏薄。
- Score：Authenticity 29.0 / Readiness 28.0 / JD Fit 26.5 / Overall 83.5

## Selection

Selected：Candidate 1（AI app product-engineering hybrid）

## Justification

- Authenticity safety first：Candidate 1 仅使用 Python、模型 API、文档问答 Demo、向量检索、多轮提示词、简单工具调用、样例测试和稳定性观察等已给素材；未新增上线、团队、指标或正式职责。
- Direct submission readiness second：最终 Delivery Resume 去除证据标签、缺口报告和占位符，采用正常简历结构，避免 ChatGPT 基线中的“待补充”和“当前不足”式自我削弱。
- JD customization third：自然引入 AI 应用、Product Engineer、需求拆解、原型验证、评估迭代、研发工具链、Agent 工程化等关键词，同时用“尝试 / 使用过 / 验证 / 关注”保持表达强度。
- Expected flip：相比 ChatGPT 基线，减少解释性长句和证据暴露，提高可投递感；相比强包装版本，保留真实性边界，目标是在真实性稳定接近满分的前提下提升 polish 与 JD fit。
