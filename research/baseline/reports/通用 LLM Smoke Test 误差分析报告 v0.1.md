# 通用 LLM Smoke Test 误差分析报告 v0.1

## 1. 报告目标

这份报告不再重复打分，而是基于已经完成的 10 个通用 LLM Smoke Test 结果，回答 5 个问题：

1. 通用 LLM 在简历生成任务中已经能做到什么？
2. 它们主要失败在哪里？
3. 哪些错误是我们产品必须避免的？
4. 哪些能力可以复用，哪些不能直接相信？
5. MVP 工作流应该因此增加哪些 guardrail？

本轮评测对象包括：

- ChatGPT
- Claude
- Gemini
- Grok
- Qwen
- Kimi
- Doubao
- GLM
- MiniMax
- DeepSeek

本轮评测样本包括：

- A2：AI 产品岗 / 素材完整 / 中风险
- B2：AI 应用开发岗 / 素材完整 / 中风险
- A6：AI 产品岗 / 素材零散 / 高风险

评测目标不是判断谁"写得最华丽"，而是判断谁更能生成一份"岗位定制、基于真实经历、接近可投递且需用户确认"的第一版实习简历。

---

## 2. 核心结论

### 结论 1：通用 LLM 已经能稳定生成"像简历"的文本

大多数模型在结构完整度和表达质量上并不差。

它们普遍能生成：

- 基本信息
- 教育背景
- 项目经历
- 技能补充
- 求职意向

所以，后续产品的差异化不能停留在"生成一份完整简历"。这个能力已经高度商品化。

如果我们的产品只是把 JD 和素材丢给模型，然后返回一份结构完整的简历，那么它和通用 LLM 的区别不大，也很难形成真正的产品价值。

---

### 结论 2：真正拉开差距的是真实性，而不是表达能力

这轮测试中，很多模型写得很顺、很像样，但问题恰好在于：

**它们太会写了。**

常见错误包括：

- 把"用过"写成"熟练使用"
- 把"尝试"写成"完成"
- 把"观察结果"写成"评估分析"
- 把"课程项目"写成"完整项目实践"
- 把"简单 demo"写成"端到端链路"
- 把"看过一些内容"写成"系统学习 / 深度理解"

这类错误不是传统意义上的完全编造，而是更危险的 **soft hallucination / 过度包装型幻觉**。

这类错误对求职简历尤其危险，因为它们看起来合理、专业、顺滑，但用户在面试中很难支撑。

---

### 结论 3：高风险样本 A6 是最有效的压力测试

A6 的特点是：

- 用户素材弱
- JD 要求强
- 用户只有零散 AI 工具使用、基础课程项目和求职准备
- 目标岗位却要求 AI 产品理解、Agent 产品体验、Prompt 基本功、AI 能力评测和产品优化

这类场景最能暴露模型是否会为了匹配岗位而越界。

结果很明显：

- ChatGPT 基本守住了边界
- GLM、DeepSeek 在 A2 / B2 上表现不错，但 A6 翻车
- 多数模型在 A6 中出现"重度用户""系统学习""多模态模型""可实习 6 个月"等未提供内容

这说明后续产品必须重点解决：

**当素材不足以支撑 JD 时，系统是否能诚实地生成弱匹配但不虚构的版本。**

如果一个系统只能在素材充分时表现好，但在素材弱、岗位强时大量越界，它不适合作为可信简历生成产品的核心。

---

### 结论 4：统一 prompt 不能彻底解决真实性问题

本轮 public input 已经明确写了：

- 不得新增未提供经历、成果、数字或职责
- 不得把课程作业包装成正式实习
- 不得把"参与""尝试""了解"写成"主导""负责""精通"
- 如果素材不足以支撑强表述，请保守表达，不要补造内容

但仍然有大量模型越界。

这说明单靠 prompt 约束不够。

产品内部必须设计独立的真实性检查、证据约束和高风险表达降级机制。

换句话说，不能把"请不要虚构"写进 prompt 就以为问题解决了。真正可靠的系统需要在生成前、生成中和生成后都进行约束。

---

### 结论 5：ChatGPT 是当前最强通用 LLM baseline，但不是最终方案

ChatGPT 是当前唯一明确通过 Smoke Test 的通用 LLM baseline：

- 平均分：81.0
- 红线样本数：0
- 三个样本均未触发严重虚构或严重误导性包装

但 ChatGPT 也不是完美方案。

例如在 A6 中，它仍然出现轻微越界，把素材中没有明确提供的"多模态模型基础了解"写进简历。

所以后续我们自己的产品至少要做到：

- 不低于 ChatGPT 的结构和表达质量
- 明显强于 ChatGPT 的证据约束
- 在 A6 这类高风险样本上更保守、更可解释
- 能告诉用户哪些内容来自原始素材，哪些地方需要用户确认

## 3. 主要错误类型

### 错误类型 1：事实新增

模型直接加入用户没有提供的信息。

典型新增内容包括：

- 项目时间
- 实习时长
- 可到岗时间
- CET-6
- GPA
- 课程名称
- 具体技术栈
- 项目成果数字
- 真实用户 / 上线 / 部署信息

这类错误最严重，因为用户如果直接投递，会有事实风险。

尤其是"可实习 6 个月及以上""CET-6""具体项目时间""具体课程名称"这类内容，看起来很普通，但如果用户没有提供，就不能写入简历。

---

### 错误类型 2：能力拔高

模型把弱证据包装成强能力。

典型模式：

- "用过 ChatGPT / Claude / Cursor" → "AI 产品重度用户"
- "对 Agent 有一些理解" → "深入理解 Agent 机制"
- "试过不同 prompt" → "具备扎实 Prompt 调优能力"
- "做过基础观察" → "完成模型效果评估"
- "看过一些 JD" → "系统研究岗位能力模型"

这类错误尤其适合我们产品作为差异化切入点。

因为用户真正需要的不是一份"看起来更强"的简历，而是一份"能被自己真实经历支撑"的简历。

---

### 错误类型 3：项目成熟度拔高

模型把个人练习、课程项目、demo 写成更成熟的项目。

典型模式：

- "简单问答 demo" → "端到端 RAG 链路"
- "工具调用实验" → "Agent 系统开发"
- "自己测几个例子" → "完整效果评测体系"
- "课程项目" → "产品探索项目 / 产品化实践"
- "基础结果观察" → "模型能力缺口分析"

这直接对应我们的产品红线：

**不能把弱项目包装成强项目。**

在实习求职中，课程项目和个人 demo 可以写，但必须诚实标注项目性质，不能写成正式业务项目、上线产品或完整工程系统。

---

### 错误类型 4：JD 关键词过拟合

模型为了贴合 JD，会主动补齐岗位关键词。

例如 JD 提到：

- Multi-Agent
- 多模态模型
- 评测体系
- Prompt 调优
- AI Native 工具链
- Agentic 行为
- AI Coding
- 研发工具链

模型就会倾向于把这些词写进简历，即使用户素材没有足够支撑。

这说明后续产品不能只做"JD → 简历"的生成链路，必须加入：

**JD 要求是否有素材证据支撑的判断。**

没有证据支撑的 JD 关键词，可以进入"学习方向 / 求职兴趣 / 待补齐能力"区域，但不能写进"项目成果 / 已具备能力 / 已完成工作"。

---

### 错误类型 5：免责声明无法抵消正文越界

很多模型会在最后写：

> 本简历基于真实素材生成，请用户确认。

但正文里仍然出现越界内容。

这说明：

**提示用户确认不是 guardrail，只是责任转移。**

我们的产品不能只在最后加一句"请确认真实性"，而要在生成前、中、后都做约束：

- 生成前：确认素材边界
- 生成中：只允许使用已映射证据
- 生成后：检查新增事实和强表述
- 交付时：标注需要用户确认的内容

---

## 4. 对 MVP 工作流的直接启发

基于本轮误差分析，MVP 不能是：

> JD + 原始素材 → 一次生成简历

更合理的流程应该是：

1. JD 解析
2. 用户素材解析
3. JD 要求与素材证据映射
4. 只允许基于已映射证据生成简历
5. 输出后做真实性检查
6. 对高风险表达降级或标注
7. 交付接近可投版，并要求用户确认

关键不是多生成一遍，而是多一个：

**证据映射层。**

如果没有证据映射层，模型会天然倾向于用 JD 语言补全简历空白。这样生成结果看起来更匹配，但真实性风险更高。

---

## 5. 产品必须设计的 guardrail

### Guardrail 1：证据引用机制

每条简历 bullet 最好能追溯到原始素材。

例如：

- 该 bullet 来自哪个素材片段
- 哪些词是模型改写
- 是否新增了未提供信息
- 是否需要用户确认

理想状态下，每条 bullet 后台都应该有证据来源。

前台不一定全部展示，但至少应该允许用户展开查看。

---

### Guardrail 2：强表述降级规则

系统需要识别并限制强词。

高风险强词包括：

- 精通
- 熟练
- 主导
- 独立负责
- 完整搭建
- 系统评测
- 深度理解
- 生产级
- 端到端
- 上线
- 高并发
- 完整链路
- 闭环优化
- 能力缺口分析

除非素材明确支持，否则自动降级为：

- 使用过
- 尝试过
- 初步了解
- 参与整理
- 做过基础观察
- 完成 demo 验证
- 对相关方向有兴趣
- 正在学习相关方法

---

### Guardrail 3：高风险样本保守模式

当系统判断"JD 要求强、用户素材弱"时，不应该强行写得匹配。

应该进入保守模式：

- 减少强结论
- 明确项目性质
- 承认经验不足
- 生成可投递但不夸大的版本
- 给用户补充建议，而不是替用户编经历

例如，用户没有系统评测经验时，不能写"搭建评测体系"，只能写：

- 做过基础结果观察
- 对不同 prompt 输出进行过简单对比
- 对模型回答质量问题有初步体感
- 后续可补充系统化评测项目

---

### Guardrail 4：JD 关键词不得无证据植入

不能因为 JD 写了 Multi-Agent、benchmark、评测体系、Agentic 行为，就自动把这些词写进简历。

系统应当先判断：

- 用户素材里是否真的有对应证据
- 证据强度是强、中、弱，还是无
- 如果没有证据，只能写成"关注 / 学习 / 希望参与"，不能写成"具备经验"

这意味着 MVP 需要一个"岗位要求—素材证据"映射表。

---

### Guardrail 5：红线表达检测

生成后必须检查：

- 是否新增事实
- 是否虚构量化结果
- 是否虚构时间
- 是否把课程项目写成实习
- 是否把 demo 写成生产系统
- 是否把"用过"写成"熟练掌握"
- 是否把"尝试"写成"完成"
- 是否把"观察"写成"评估体系"
- 是否把"关注"写成"深入理解"

一旦命中红线，系统应该自动改写，而不是直接交付给用户。

---

## 6. 对后续产品设计的判断

这轮结果实际上证明了我们的产品机会点是成立的。

因为如果通用 LLM 已经完美解决问题，我们就没有必要做新产品。

但现在看到的是：

- 它们会写
- 它们懂 JD
- 它们表达能力强
- 但它们不稳定
- 它们容易为了匹配而越界
- 它们不能天然保证证据约束

所以我们的产品不应该和通用 LLM 比"谁更会写"，而应该比：

**谁更能在真实性边界内生成岗位定制简历。**

这正好回到 Problem Definition 的核心：

> 不是 AI 改简历，而是基于真实素材的受约束第一版生成。

---

## 7. 对 MVP 的明确要求

基于本轮误差分析，MVP 至少需要满足以下要求：

### 1. 不能只做单轮生成

单轮生成已经被通用 LLM 覆盖，而且容易越界。

MVP 应该是：

> 解析 → 映射 → 生成 → 校验 → 修订 → 用户确认

而不是：

> 输入 → 生成

---

### 2. 必须有证据映射层

在生成简历前，系统需要先输出内部结构：

- JD 关键要求
- 用户素材证据
- 证据强度
- 可写入简历的表达
- 不可写入简历的要求

这层结构决定了产品是否可信。

---

### 3. 必须支持高风险降级

当素材不足时，系统不应该强行匹配 JD。

它应该给出更保守的写法，并告诉用户：

- 哪些岗位要求目前缺少证据
- 哪些经历只能弱表达
- 哪些能力建议补项目再写
- 哪些内容不能写进简历

---

### 4. 必须把"用户确认"做成流程，而不是免责声明

用户确认不应该只是最后一句话。

它应该体现在：

- 哪些内容需要用户确认
- 哪些内容证据不足
- 哪些内容建议补充
- 哪些内容已被系统降级处理

这样用户才知道自己到底在确认什么。

---

## 8. 下一步建议

基于本轮结果，下一步有两个方向：

### 方向 A：继续跑 A/C/D 类工具

继续测试：

- Jobscan
- WonderCV
- BOSS 直聘简历
- Oaki
- Rezi
- Teal
- 全民简历
- Resume Matcher
- resume-tailoring-skill
- smart-resume-matcher
- claude-code-job-tailor
- Resume Optimizer AI
- JobMatchAI

这个方向可以补完整个 baseline。

### 方向 B：基于误差分析设计 MVP 工作流 v0.1

开始设计我们自己的系统：

- JD 解析
- 素材解析
- 证据映射
- 受约束生成
- 真实性检查
- 高风险降级
- 用户确认

当前建议优先选择方向 B。

原因是：通用 LLM 的误差已经足够指导第一版系统设计。继续跑更多工具当然有价值，但如果不先把 MVP 工作流设计出来，后续评测会停留在机械比较层面，无法转化为产品方案。

---

## 9. 阶段性结论

本轮通用 LLM Smoke Test 的最大收获是：

**"会生成简历"已经不是稀缺能力，"在真实性边界内生成岗位定制简历"才是稀缺能力。**

这意味着我们的产品机会点应该进一步收敛为：

> 做一个基于 JD 与原始素材证据映射的、受约束的第一版简历生成系统。

后续 MVP 不应该追求"写得更漂亮"，而应该追求：

- 更少虚构
- 更少越界包装
- 更清楚的证据来源
- 更可控的强表述
- 更低的用户修改成本
- 更可信的接近可投版输出
