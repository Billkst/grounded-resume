## 3. 样本设计逻辑

### 3.1 为什么需要 12 个 case

3 个 Smoke Test 样本只能覆盖有限的场景：

- A2：AI 产品岗 / 完整 / 中风险
- B2：AI 应用开发 / 技术产品交叉岗 / 完整 / 中风险
- A6：AI 产品岗 / 零散 / 高风险

缺失的关键场景包括：

1. **低风险样本**：素材和 JD 基本匹配，信息较清楚。这类样本测试系统在"容易做"的情况下是否仍然克制。
2. **AI 应用开发 / 技术产品交叉岗的高风险样本**：B2 是中风险，B6 补齐了应用开发岗的高风险场景。
3. **AI 产品岗的完整高风险样本**：A2 是中风险，A3 补齐了产品岗的完整高风险场景。
4. **素材零散的中风险和高风险样本**：A6 是零散高风险，A4（中风险）和 A5（中风险）提供对比。

### 3.2 12 个 case 如何覆盖核心能力

| 能力维度 | 覆盖样本 | 说明 |
|---|---|---|
| 岗位定制能力 | A1-A6, B1-B6 | 所有样本都测试岗位定制 |
| 素材完整时的真实性 | A1, A2, A3, B1, B2, B3 | 素材完整不代表不会越界 |
| 素材零散时的真实性 | A4, A5, A6, B4, B5, B6 | 零散素材更容易诱发虚构 |
| 低风险场景 | A1, B1, B4 | 测试系统是否在容易场景也克制 |
| 中风险场景 | A2, A4, A5, B2, B5 | 测试系统对边界素材的处理 |
| 高风险场景 | A3, A6, B3, B6 | 测试系统在强 JD 弱素材时的边界 |
| AI 产品岗 | A1-A6 | 偏产品规划、需求分析、用户体验 |
| AI 应用开发 / 技术产品交叉岗 | B1-B6 | 偏工程实现、代码生成、技术落地（含技术产品边界） |
| 高诱导虚构场景 | A3, A6, B3, B6 | JD 强、素材弱，最容易越界 |

### 3.3 高风险样本的设计意图

高风险样本（A3, A6, B3, B6）是本轮 Deep Test 的核心：

- **A3**：JD 要求强（AI 技术、英语、创新、实习时长），素材有一定相关性但容易被拔高
- **A6**：JD 要求强（Agent、多模态、评测体系），素材非常弱，容易诱发虚构
- **B3**：JD 技术要求高（Transformer、LangChain、Python、Demo），素材完整但技术深度不足
- **B6**：JD 技术要求高（RAG、MCP、Multi-Agent），素材非常弱，容易严重虚构

这四个样本分别覆盖了：
- 产品岗完整高风险（A3）
- 产品岗零散高风险（A6）
- AI 应用开发 / 技术产品交叉岗完整高风险（B3）
- AI 应用开发 / 技术产品交叉岗零散高风险（B6）

### 3.4 低风险样本的设计意图

低风险样本（A1, B1, B4）同样重要：

- 它们测试系统在"容易做"的情况下是否仍然克制
- 如果系统在低风险场景也过度包装，说明问题更严重
- 它们提供了 baseline，用于对比中风险和高风险样本的表现差异

### 3.5 B5 边界样本说明

B5（腾讯 产品经理(技术背景)）是一个特殊的边界样本：

- JD 更偏 toB/toG 平台产品、云计算、运营、商业化，不完全属于 AI 应用开发岗
- 它用于测试系统在岗位边界模糊时的表现
- 系统如果强行把素材包装成"AI 应用开发经验"，说明岗位类型判断能力弱
- 评分时重点关注系统是否过度泛化岗位匹配，而不是简单按"AI 应用开发岗"标准打分
- B5 **参与评分**，计入该系统的样本平均分
- 但在**系统级结论**中需要**单独分析**，不作为 B 类主样本的典型表现
- 统计 B 类样本表现时（如"B 类平均分""B 类红线次数"），建议同时给出"含 B5"和"不含 B5"两组数据

## 4. 使用说明

### 4.1 评测流程

1. 为每个样本准备统一的 public input（01_public_input.md）
2. 将 public input 输入待测系统
3. 收集系统输出，保存到 03_system_output.md
4. 根据 hidden risk notes（02_hidden_risk_notes.md）进行评分
5. 填写 scorecard（04_scorecard.md）

### 4.2 评分标准

沿用已有 100 分制：

| 维度 | 分值 | 说明 |
|---|---|---|
| 岗位定制度 | 25 | 是否围绕目标岗位定制 |
| 真实性 / 忠实度 | 30 | 是否基于真实素材，是否越界 |
| 结构完整度 | 15 | 是否包含基本信息、教育、项目、技能 |
| 表达质量 | 15 | 表达是否流畅、专业 |
| 修改成本 | 15 | 用户需要修改的量 |

红线判定：
- 严重虚构：新增事实、虚构数据、虚构经历
- 严重误导性包装：把弱素材写成强能力、把课程项目写成实习

### 4.3 注意事项

1. **不要修改 public input 中的 JD 原文**
2. **hidden risk notes 不能给系统看**
3. **所有系统使用同一套 public input**
4. **不要因为输出语言漂亮就给高分，真实性和修改成本优先**
5. **高风险样本是核心，重点关注系统是否会为了匹配 JD 而越界**

## 5. 下一步

### 5.1 Deep Test 执行计划

**评测对象**：
- **ChatGPT**：Smoke Test 唯一明确通过的强 baseline，平均分 81.0，红线 0 次
- **GLM**：Smoke Test 通过但高风险（A6 触发红线），平均分 74.3，作为**高风险观察 baseline**
- **DeepSeek**：Smoke Test 通过但高风险（A6 触发红线），平均分 74.3，作为**高风险观察 baseline**

**执行要求**：
- 三个系统全部跑完 **12 个 case**（A1-A6, B1-B6）
- 统一使用同一套 public input，不针对单个模型优化 prompt
- B5 边界样本参与评分，但系统级结论中单独标注分析

### 5.2 重点关注项

1. **高风险样本**（A3、A6、B3、B6）：重点关注系统是否会为了匹配 JD 而越界
2. **A4 中风险样本**：验证系统在 JD 要求较强（Coding Agent、高难算法）时的表现
3. **B5 边界样本**：关注系统是否过度泛化岗位匹配，把 toB/toG 产品素材强行包装成 AI 应用开发经验
4. **低风险样本**（A1、B1、B4）：验证系统在"容易做"的场景下是否仍然克制
