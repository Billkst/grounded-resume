# 仓库结构重构方案 v0.1

> **目标**：将当前 research-heavy 仓库整理成适合后续 MVP 开发的 research + product + code 项目结构。  
> **范围**：仅规划，不移动文件。  
> **适用阶段**：Baseline 已完成，即将进入 MVP 工作流设计与代码开发。

---

## 1. 当前目录问题诊断

### 1.1 根目录扁平化，同类目录散落各处

当前根目录下存在 `outputs/`、`outputs_deep/`、`outputs_c_tools/`、`outputs_d_repos/`、`outputs_a_ats/`、`scorecards/`、`scorecards_deep/`、`scorecards_c_tools/` 等多个同类目录，命名风格不统一（有的带下划线、有的不带），且全部平铺在根目录，导致仓库顶层混乱。

### 1.2 研究数据与方法论文档混放

- `docs/` 中同时存放了**方法论文档**（如 `Baseline 评测框架 v0.1.md`）和**业务参考文档**（如 `AI产品经理招聘要求.md`）。
- `smoke_test/` 中每个样本包含 4 份文件（`01_public_input.md`、`02_hidden_risk_notes.md`、`03_system_output.md`、`04_scorecard.md`），其中后两份是系统输出和评分，本质上属于**评测结果**，而非纯输入数据。

### 1.3 缺少产品文档专属目录

Baseline 阶段产出了大量执行方案、范围决策、综合分析报告，但缺少一个统一的 `product/` 目录来存放 PRD、工作流设计、技术规格、架构决策（ADR）等产品级文档。

### 1.4 完全没有代码目录

仓库当前是 100% 文档与数据，没有任何 `src/`、`tests/`、`scripts/` 等代码基础设施目录。下一阶段进入 MVP 开发后，必须有清晰的代码组织空间。

### 1.5 缺少测试与评测基础设施

当前评测数据散落在各处，但没有统一的 `tests/` 或 `evals/` 目录来存放测试固件、回归测试、评测基准脚本。

### 1.6 内部链接潜在失效风险

部分文档使用了类 Wiki 链接（如 `[[问题定义 v1.0]]`），大规模迁移后如果不更新，内部导航会失效。

## 6. 后续 MVP 代码目录建议

基于《Baseline 综合分析报告 v1.0》中对 MVP 的 9 项要求，建议 `src/core/` 按以下模块组织：

| 模块名 | 路径 | 对应 MVP 要求 | 职责 |
|---|---|---|---|
| JD 解析层 | `src/core/jd-parser/` | 要求 1 | 提取岗位要求、技能、经验年限、硬条件 |
| 素材解析层 | `src/core/material-parser/` | 要求 2 | 从用户零散素材中抽取结构化经历事实 |
| 证据映射层 | `src/core/evidence-mapper/` | 要求 3 | 建立 JD 要求与用户素材之间的映射，识别匹配项和缺口 |
| 受约束生成层 | `src/core/constrained-generator/` | 要求 4 | 在真实性约束下生成简历，禁止无证据植入 |
| 表述降级器 | `src/core/expression-guard/` | 要求 5 | 对弱素材使用保守表达，而非包装成强经历 |
| 红线检测器 | `src/core/redline-detector/` | 要求 6 | 识别并阻止高风险表达（虚构技能、夸大职责、推断指标） |
| 保守模式 | `src/core/conservative-mode/` | 要求 7 | 当 JD 与素材差距大时，采用保守策略 + gap 披露 |
| 用户确认层 | `src/core/user-confirmation/` | 要求 8 | 展示每条内容的证据来源，支持用户修改或拒绝 |
| 简历格式化器 | `src/core/resume-formatter/` | 要求 9 | 中文实习简历场景的结构、表达风格和素材采集策略适配 |

此外：

- `src/pipelines/`：按工作流编排上述模块，如 `generate-resume-pipeline.ts`。
- `src/models/`：统一定义数据 Schema，如 `JDParseResult`、`MaterialFact`、`EvidenceMapping`、`ResumeSection`。
- `src/api/`：对外暴露 CLI 或 Web API。
- `tests/evals/`：存放回归测试脚本，确保每次代码变更后 Baseline 能力不退化。

## 7. 风险与注意事项

### 7.1 Git 历史保持

- 迁移文件时务必使用 `git mv` 而非直接 `mv`，以保持文件的版本历史可追溯。
- 如果一次迁移文件过多，建议分批次提交，每批一个独立 commit，commit message 清晰标注迁移范围。

### 7.2 内部链接失效

- 文档中存在的类 Wiki 链接（如 `[[问题定义 v1.0]]`）在文件移动后可能失效。
- 迁移完成后，需全局搜索 `[[` 和 `]]` 模式，更新为新的相对路径或保持链接文本但更新指向。

### 7.3 路径引用检查

- 部分文档可能在正文内嵌了文件路径引用（如 `"见 outputs/chatgpt/A2_output.md"`）。
- 迁移后需全文检索旧路径关键词（如 `outputs/`、`scorecards/`、`smoke_test/`），批量更新为新路径。

### 7.4 大文件批量迁移

- `outputs/`、`outputs_deep/`、`scorecards/` 等目录下文件数量众多（约 200+ 个文件）。
- 建议使用脚本批量执行 `git mv`，避免手动遗漏。
- 可先在 `scripts/data-migration/` 下编写一次性迁移脚本，执行后归档该脚本。

### 7.5 并行协作冲突

- 如果重构期间有其他协作者同时提交文档或数据，可能产生路径冲突。
- 建议：
  1. 选择一个协作低峰期执行迁移；
  2. 提前在 README 或群聊中公告重构计划；
  3. 迁移完成后立即推送，并通知全员更新本地分支。

### 7.6 目录深度增加

- 新结构最深可达 6 层（如 `research/baseline/smoke-test/outputs/llm/chatgpt/A2_output.md`）。
- 需确保团队成员的 IDE、文件管理器、命令行工具能正常处理该深度，无路径长度限制问题（Windows 默认 260 字符限制需注意）。

### 7.7 空目录占位

- `src/`、`tests/`、`product/designs/` 等新目录在初始阶段为空。
- 建议在空目录下放置 `.gitkeep` 或 `README.md` 占位文件，确保目录能被 Git 追踪。

### 7.8 迁移后验证清单

迁移完成后，建议执行以下验证：

- [ ] `git status` 无未追踪的旧路径文件残留。
- [ ] `git log --follow <关键文件>` 确认历史保留。
- [ ] 全局搜索旧路径关键词，确认无硬编码引用。
- [ ] 打开 3~5 份关键文档，验证内部链接可正常跳转。
- [ ] 运行 `tree -L 3` 检查新结构是否符合预期。
