# Baseline 综合分析报告 v1.0

## 1. 报告目标

本报告用于总结 B/C/D/A 四类 baseline 的测试与分析结果，回答一个核心问题：

> 现有方案是否已经解决"基于目标岗位 JD 和用户原始素材，生成一份岗位定制、基于真实经历、接近可投递且需用户确认的第一版实习简历"这个问题？

## 2. Baseline 分类

| 类别 | 名称 | 代表对象 | 本阶段处理方式 |
|---|---|---|---|
| A 类 | 规则型 / ATS 工具 | Jobscan、WonderCV 关键词扫描、BOSS 诊断 | 范围收敛，不实测 |
| B 类 | 通用 LLM | ChatGPT、Claude、Gemini、Grok、Qwen、Kimi、Doubao、GLM、MiniMax、DeepSeek | Smoke + Deep Test |
| C 类 | 专用 AI 简历工具 | WonderCV、全民简历、BOSS 简历、Rezi、Teal、Oaki | 中文工具 Smoke Test，英文工具暂缓 |
| D 类 | 开源项目 | Resume Matcher、Smart Resume Matcher、Claude Code Job Tailor 等 | Repo Capability Review，不运行 |

## 3. B 类结论：通用 LLM

### 3.1 测试范围

B 类完成了：

- 10 个通用 LLM Smoke Test
- ChatGPT / GLM / DeepSeek Deep Test
- 30 条 Smoke Test 输出
- 36 条 Deep Test 输出

### 3.2 核心结论

**ChatGPT 是当前唯一强 baseline。**

GLM 和 DeepSeek 在 Smoke Test 中进入高风险观察，但 Deep Test 证明二者在真实性和高风险样本上不稳定。

### 3.3 关键发现

通用 LLM 已经能生成结构完整、表达自然的简历，但核心问题是：

- **容易新增事实**：模型会在用户未提供的素材基础上补充技能、工具、项目细节；
- **容易把弱素材写成强经历**：一段简单的课程作业可能被写成"独立负责的大型项目"；
- **容易为了匹配 JD 植入无证据关键词**：用户未提及的技能可能因 JD 要求而被写入简历；
- **单靠 prompt 无法稳定解决真实性问题**：即使有"不要虚构"的指令，不同模型、不同样本的遵循程度差异很大；
- **高风险样本最能暴露模型越界**：当 JD 要求与用户素材差距较大时，模型更容易"硬凑"经历来满足匹配。

### 3.4 对本项目的启发

本项目不能和通用 LLM 比"谁更会写"，而应该比：

> 谁更能在真实性边界内，把 JD 要求和用户真实素材映射成可信简历。

这意味着本项目的核心竞争力不是"生成质量"，而是：

1. **证据映射**：每条简历内容都能找到对应的用户素材证据；
2. **强约束生成**：禁止无证据植入、禁止夸大、禁止新增事实；
3. **保守模式**：对高风险素材使用保守表达，而非包装成强经历；
4. **用户确认**：让用户能看到每条内容的证据来源，并有权修改或拒绝。

## 4. C 类结论：专用 AI 简历 / 求职工具

### 4.1 测试范围

本轮实际测试中文工具：

- WonderCV / ClawCV
- 全民简历
- BOSS 直聘简历

英文工具暂缓：

- Rezi
- Teal
- Oaki

### 4.2 核心结论

**C 类中文工具没有产生强 baseline。**

- **WonderCV**：偏已有简历 + JD 分析 / 优化，不支持从零生成；
- **全民简历**：偏结构化表单 AI 生成，输出模板化，JD 定制能力弱；
- **BOSS 简历**：可生成完整简历，但严重虚构和过度包装，真实性风险高；
- **英文工具**：因市场和语言场景不同，暂不纳入中文 MVP v0.1 对标。

### 4.3 关键发现

专用 AI 简历工具并不等于解决本项目核心任务。

很多工具实际解决的是：

- 已有简历优化
- 模板化生成
- 结构化字段填充
- ATS / 关键词改写
- 英文 resume 生成

而不是：

> JD + 原始素材包 → 证据约束第一版简历

C 类工具的共性问题是：

1. 要么依赖已有简历（WonderCV）；
2. 要么模板化严重、JD 定制弱（全民简历）；
3. 要么虚构风险高（BOSS 简历）；
4. 要么面向英文市场（Rezi / Teal / Oaki）。

## 5. D 类结论：开源项目

### 5.1 评估范围

评估了 6 个 GitHub 开源项目：

- Resume Matcher
- resume-tailoring-skill
- smart-resume-matcher
- claude-code-job-tailor
- Resume Optimizer AI
- JobMatchAI

### 5.2 核心结论

**6 个项目全部依赖已有简历，没有一个支持"零散原始素材 → 第一版实习简历"。**

### 5.3 主流路线

D 类项目主要集中在：

- 已有简历 + JD 匹配分析
- 已有简历 + JD tailoring
- 简历 bullet rewrite
- ATS / keyword scoring
- 浏览器插件自动填表
- Claude Code workflow

### 5.4 保留价值

虽然不作为直接 baseline，但 D 类项目对实现有参考价值：

| 项目 | 保留价值 |
|---|---|
| **Resume Matcher** | diff preview、confirm、PDF 导出、真实性治理（diff allowlist、preview hash、keyword gap analysis） |
| **Smart Resume Matcher** | 结构化简历 JSON schema、accept/reject/undo、side-by-side comparison、Zod schema |
| **Claude Code Job Tailor** | YAML 素材库、weighted scoring、job focus array、validation-first workflow |
| **Resume Tailoring Skill** | truthfulness 核心原则、confidence scoring、gap disclosure、branching discovery、checkpoint |
| **JobMatchAI** | 浏览器端 JD 抽取、profile 解析、bullet 级用户确认、DOCX 非破坏式生成 |
| **Resume Optimizer AI** | 真实性风险反例（prompt 允许推断指标） |

## 6. A 类结论：规则型 / ATS 工具

### 6.1 核心结论

**A 类工具不进入实测。**

原因是 A 类工具主要解决：

> 已有简历 + JD → 匹配检查 / ATS 评分 / 关键词优化

而本项目解决：

> JD + 原始素材 → 第一版可信简历

任务形态不同。

### 6.2 参考价值

A 类可以作为后续产品中的辅助模块参考：

- 关键词提取
- 硬条件匹配
- 缺失技能提示
- 投递前检查
- ATS-friendly 格式检查

但不适合作为第一版生成系统的直接 baseline。

## 7. 四类 baseline 横向对比

| 类别 | 是否支持 JD | 是否支持原始素材 | 是否从零生成 | 是否有真实性约束 | 是否强 baseline |
|---|---|---|---|---|---|
| A 类 ATS 工具 | 是 | 否 | 否 | 弱 | 否 |
| B 类通用 LLM | 是 | 是 | 是 | 不稳定 | ChatGPT 勉强可作为强 baseline |

## 8. 最终结论

**现有方案没有完整解决本项目核心任务。**

本项目机会点成立：

> 面向中文实习求职者，做一个基于 JD 与原始素材证据映射的、受约束的第一版简历生成系统。

现有方案的共同缺口包括：

1. **依赖已有简历**：B/C/D/A 类中，只有 B 类通用 LLM 支持从零生成，但真实性不稳定；其余全部要求已有简历。
2. **不支持零散素材到第一版简历**：没有系统支持"用户给零散素材，系统生成第一版完整简历"。
3. **缺少 JD 要求与素材证据映射**：现有方案大多把 JD 作为 prompt 上下文或关键词来源，缺少"JD 要求 → 用户经历证据 → 缺口分析 → 用户补充"的闭环映射。
4. **缺少逐条 bullet 证据追溯**：没有项目支持"每条简历内容都能追溯到原始素材片段"。
5. **缺少强表述降级**：现有方案倾向于"包装"和"优化"，而非"保守表达"和"降级"。
6. **缺少高风险样本下的保守生成机制**：当 JD 要求与用户素材差距较大时，现有方案要么硬凑、要么放弃，缺少"保守生成 + gap 披露"的中间态。
7. **中文实习场景支持不足**：现有方案要么面向英文市场，要么面向通用职场简历，缺少针对中文实习场景的专门设计。

## 9. 对 MVP 的要求

基于 baseline 结论，MVP 必须具备：

1. **JD 解析层**：提取岗位要求、技能、经验年限、硬条件；
2. **用户素材解析层**：从用户提供的零散素材中抽取结构化经历事实；
3. **JD 要求—素材证据映射层**：建立 JD 要求与用户素材之间的映射关系，识别匹配项和缺口；
4. **受约束生成层**：在真实性约束下生成简历，禁止无证据植入、禁止夸大、禁止新增事实；
5. **强表述降级规则**：对弱素材使用保守表达，而非包装成强经历；
6. **红线表达检测**：识别并阻止高风险表达（如虚构技能、夸大职责、推断指标）；
7. **高风险样本保守模式**：当 JD 要求与素材差距较大时，采用保守生成策略 + gap 披露；
8. **用户确认与证据查看机制**：让用户能看到每条内容的证据来源，并有权修改或拒绝；
9. **中文实习简历场景适配**：针对学生/实习/校招场景设计简历结构、表达风格和素材采集策略。
