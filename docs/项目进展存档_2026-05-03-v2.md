## 一、项目背景与问题

### 1.1 当前产品状态
- **产品名称**: grounded-resume
- **当前阶段**: MVP 已成形，生成策略通过三轮盲评验证
- **核心问题**: `ConstrainedGenerator` 是规则引擎（关键词替换），生成简历只是用户原话的简单改写
- **当前突破**: 第三轮 Jobs v3 策略已全面领先 ChatGPT 基线，但尚未工程化落地到产品代码中

### 1.2 产品哲学演进

| 阶段 | 哲学 | 结果 |
|---|---|---|
| Baseline | 真实性为底线，100%素材溯源，禁止虚构 | 简历过于保守，竞争力不足（ChatGPT 81.0） |
| Jobs v1 | 完美简历+允许虚构+分离式可信度报告 | 被识别为虚构，惨败（66.6 vs 81.2） |
| Jobs v2 | 保守推断+可信度标注+Gap Report | 险胜（83.13 vs 82.59），A6 稀疏素材失败 |
| **Jobs v3** | **输出分离+素材密度自适应+Chain-of-Density** | **大胜（93.07 vs 86.27），全面领先** |

## 三、专家诊断与策略升级

本轮邀请了 3 位专家代理并行诊断：

### 3.1 产品大师视角（Steve Jobs / 雷军）

**核心瓶颈**:
1. 产品定义太窄 — 用户在买"投递结果"，不是"简历草稿"（严重 9/10）
2. 标注语和 Gap Report 破坏了"开箱即用"（严重 10/10）
3. A6 稀疏素材不是生成问题，而是"冷启动失败"（严重 9/10）

**策略建议**:
- 产品从"简历生成器"升级为"求职竞争力启动器"
- 输出拆成两层：用户可直接投递层 + 编辑辅助层
- 为稀疏素材设计 Resume Bootstrap Mode（最低可投递版 + 5 分钟补全问答 + 48 小时补强任务）

### 3.2 技术架构视角

**核心瓶颈**:
1. `ConstrainedGenerator` 是规则改写器，不是生成系统（严重 10/10）
2. LangGraph 是线性管道，不是智能编排（严重 9/10）
3. 输出物耦合：简历正文、标签、Gap、可信度没有架构级分离（严重 9/10）

**架构建议**:
- 从"单阶段生成"升级为"编译器式多阶段生成流水线"
- 引入 `MaterialRichnessClassifier`（素材丰富度分类）
- 引入 `ClaimGraphBuilder`（claim-level 证据链）
- 输出物必须分离：投递版 / 内部版 / Gap Report / Evidence Report

### 3.3 AI 专家视角（Karpathy）

**核心瓶颈**:
1. 把"不要幻觉"交给自然语言 Prompt 控制（严重 9/10）
2. 缺少 claim-level evidence tracking（严重 10/10）
3. 保守 Prompt 把稀疏素材打成"低信息密度输出"（严重 8/10）

**算法建议**:
- 从 "Prompt → Resume" 改为 "Evidence → Claims → Verified Bullets → Resume"
- 引入 Generator + Verifier 双模型机制
- 稀疏素材用 Chain-of-Density，不是简单保守
- 数字 claim 用程序硬校验，不交给 LLM
- 多候选生成 + reranking

## 四、已确立的完美简历标准

### 4.1 内容质量

## 五、第三轮策略详解（当前最佳实践）

### 5.1 输出分离
每个样本产出 5 类文件：
1. **Delivery Resume**: 干净、无标签、可直接投递
2. **Internal Review Resume**: 带证据标签，供用户确认
3. **Gap Report**: 单独文档，列出缺口和补齐建议
4. **Claim Ledger**: JSON，逐条 bullet 拆分 claim 并绑定证据
5. **Candidate Rerank**: 记录 3-5 个候选版本和选择原因

### 5.2 素材密度自适应

| 密度 | 样本 | 策略 | 关键特征 |
|---|---|---|---|
| Rich | A2 | Achievement-focused evidence amplification | 允许标准表达和中等推断，优先生成 STAR bullet |
| Moderate | B2 | Balanced generation | 简洁成就 bullet，避免过度声称成果 |
| Sparse | A6 | Chain-of-Density | 三阶段流水线：factual draft → domain specificity → business relevance |

### 5.3 Chain-of-Density（稀疏素材专用）

**目标**: 让稀疏素材听起来专业且可信，同时不编造数字和成果。

**Pipeline**:
1. Factual draft: 只用字面事实写 bullet
2. Domain specificity pass: 添加安全的 AI/product/process 名词
3. Business relevance pass: 连接业务相关性但不声称结果
4. Safety pass: 移除所有 unsupported number、ownership verb、outcome claim

**示例**:
- Weak: "Helped prepare reports."
- Better: "Prepared recurring operational reports that consolidated key updates for stakeholders and improved visibility into weekly priorities."

### 5.4 推断控制硬规则

- 原素材没有数字时，不得出现百分比、营收、用户数、团队人数、准确率、转化率、效率提升
- 原素材只有"参与/协助/负责部分"时，不得写"主导/统筹/独立负责/owner/lead"
- 原素材没有上线、落地、商业结果时，不得写"上线后提升""支撑业务增长"
- 原素材没有真实公司实习时，不得伪造公司实习经历
- A6 sparse 样本默认禁用所有新增数字与强 ownership 动词

## 八、关键认知更新

### 8.1 之前的错误认知 → 现在的正确认知

| 错误认知 | 正确认知 |
|---|---|
| "允许虚构+分离式报告标注"就能解决素材不足 | 简历正文本身必须看起来像真的，推断要有逻辑依据；缺失能力用 Gap 报告指出 |
| 一份带标签的简历草稿就是产品输出 | 用户要的是可直接投递的简历 + 内部确认界面 + 提升指南 |
| 所有素材用同一套生成策略 | Rich/Moderate/Sparse 需要完全不同的生成策略 |
| Prompt 约束就能防止幻觉 | 需要系统级约束：Evidence→Claim→Verify→Resume |

### 8.2 产品的真正价值

不是"帮用户编造一份看起来很强的简历"，而是：
1. **基于现有素材，帮用户看到"最好的自己"长什么样**
2. **诚实告知"要达到最好的自己，你还缺什么"**
3. **提供具体可行的补齐路径**
